# Phase 15：资源编码 / 映射语义 / 链路监控收口

本轮调整的目标不是继续扩字段，而是降低认知负担，并把前后端术语与行为统一到更自洽的模型上：

## 1. 术语收口
- UI 层统一使用“编码”而不是“DeviceID / 设备编号 / 资源 ID”混称。
- 本地资源、隧道映射、节点与隧道、链路监控四个页面分别承担不同职责。
- 本端编码 / 对端编码 / 资源编码统一要求为 20 位国标编码。

## 2. 本地资源
- “资源 ID” 与 “DeviceID” 合并为单一字段：资源编码。
- 增加资源类型字段，默认按服务类型建模；若未来接摄像机或其他资源，可继续扩展。
- 允许方法改成下拉多选。
- 最大内联响应体 / 最大请求体 / 最大响应体改成自动推导，只读展示，不再由单条资源手工输入。

## 3. 隧道映射
- 映射页改为以“下级资源”为基础的总览页。
- 用户只补充本地监听信息（监听 IP / 端口 / 路径），不再在此页维护对端目标 IP/端口和单条超时。
- 对端目标、单条连接超时、请求超时、响应头超时继续由兼容层承载，但 UI 不再把这些字段暴露给主流程。

## 4. 链路监控
- 移除资源数量 / 映射摘要这类资源配置相关信息。
- 仅保留注册状态、心跳状态、已注册对端、待完成会话、入站会话等运行态指标。

## 5. 后端协同
- NodeConfigEndpoint 增加 node_type。
- TunnelMapping 增加 resource_type，并补了 20 位国标编码校验。
- 本地资源列表 / 隧道映射总览补充 resource_code / resource_type 等字段。
- 节点与隧道 / 节点配置保存链路增加 20 位国标编码校验。
