# 全盘 Review / 复盘与清洁结论（2026-03-18）

本文档记录对源码、UI、文档、脚本的一次全盘 Review 结论，目标是把工程保持在**可维护、可验收、可复盘**的状态，并为后续改动建立固定原则。

## 一、当前确认的主问题

### 1. 术语与页面职责曾发生漂移

已观察到：

- README / docs 索引仍残留“节点与隧道”旧名称
- 历史阶段文档与当前 UI 主术语不一致
- 本地资源、本地监听端口、隧道映射曾出现职责交叉

结论：

- 当前主线必须固定为“节点与级联 / 本地资源 / 隧道映射 / 链路监控”四分结构
- 历史 phase 文档只能作为归档参考，不再作为当前事实来源

### 2. 配置职责曾双写/影子兼容

已观察到：

- `server.port` 历史兼容残留
- 本地资源与隧道映射曾共享/隐式生成同一份监听信息
- `config.yaml` 与运行态业务数据存在职责混淆风险

结论：

- `ui.listen_port` 是管理面端口唯一来源
- 本地资源与隧道映射必须拆开
- 运行态业务配置只走 workspace/resources/mappings API

### 3. 传输链路的完成条件与性能口径不够稳定

从运行日志可确认：

- 某些 RTP 请求在 `response_end_received` 之后仍等待额外 idle/buffer 窗口才真正结束
- Web 页面首屏性能主要受多资源并发 + RTP 模式完成条件影响
- 大文件下载受整体客户端 timeout / 本地 WriteTimeout / RTP 接收完成条件多重影响

结论：

- RTP 链路必须按“已知长度 / ResponseEnd / BYE”组合完成
- 不能用“整包缓存 + 固定 idle 等待”方式承载页面与大文件

### 4. 访问日志口径不够利于入口分析

已观察到：

- 浏览器页面会触发一串子请求
- 如果全部逐条记录，会严重稀释入口路径分析
- 当前日志虽然有 `mapping-runtime`，但对入口聚合、transport 模式和阶段耗时还不够稳定

结论：

- 访问日志必须按入口路径聚合为主
- 子请求成功日志需要抑制或聚合
- 性能日志必须补齐 transport 阶段指标


### 5. 启动摘要 / 总览监控 / 访问日志口径仍需统一

从运行日志与启动摘要可以看到：

- 启动摘要仍可能输出 `business_execution: protocol_only`，但本地资源 + 目录同步 + 远端监听补充后，隧道请求已经可以执行
- 总览监控当前偏“是否成功”，缺少 transport 阶段耗时
- 访问日志仍需更明确地区分“入口访问”与“浏览器子资源请求”

结论：

- `business_execution` 需要重新定义为：
  - 协议层是否可用
  - 本地资源发布是否可用
  - 隧道映射执行是否可用
- 总览监控需要补三类指标：
  - INVITE 建链耗时
  - upstream HTTP 耗时
  - ResponseEnd 到本地落地耗时
- 访问日志需要以入口路径为主口径，并对同源子请求做聚合/抑制

## 三、建议固定的治理动作

### 3.1 每次改菜单或页面职责时

必须同步改：

- `README.md`
- `docs/README.md`
- UI 路由与导航
- 至少一个静态一致性检查脚本

### 3.2 每次改配置字段时

必须同步改：

- config example
- 后端解析
- UI 表单与类型
- 当前有效文档
- 验收脚本/一致性脚本

### 3.3 每次改 transport/link chain 时

必须同步改：

- 处理链路日志
- 性能指标
- 抓包核验清单
- 诊断导出说明

## 五、仍需继续跟踪的运行问题

### P0

- RTP 响应完成后仍存在额外等待窗口的边界情况
- Web 页面多资源并发时的首屏耗时仍需持续压缩
- 大文件在更大体量和更慢网络下的完整性还需压测验证

### P1

- 访问日志入口聚合还可进一步增强（如 session/page 维度）
- 总览监控还应补 transport 阶段耗时指标，而不是只显示粗粒度成功/失败
- 目录同步与映射执行的观测口径仍可继续统一
